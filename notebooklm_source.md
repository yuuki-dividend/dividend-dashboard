# 「配当ちゃん」ができるまで ── AIと4日間で作った高配当株ダッシュボード

## はじめに：このドキュメントについて

このドキュメントは、プログラミング経験ゼロの個人投資家が、AIアシスタント（Claude）との対話だけで、本格的な投資分析ツールを4日間で作り上げたプロセスを記録したものです。スライド作成の素材として、制作の背景・指示の出し方・改善の経緯を時系列で解説しています。

---

## 第1章：制作の背景 ── なぜこのツールが必要だったのか

### 課題：高配当株投資の情報管理が大変

高配当株投資では、一般的に50銘柄以上を分散して保有します。しかし、各銘柄の配当利回り・配当性向・増配履歴・業績トレンドなどを毎日チェックするのは非常に手間がかかります。

従来は以下のような方法で管理していました：
- マネーフォワードで保有銘柄の時価を確認
- Yahoo!ファイナンスや株探で個別に業績を調べる
- Googleスプレッドシートに手動で数値を転記
- 「買い時かどうか」の判断は感覚に頼る部分が多い

**問題点**：情報が分散しており、全体像が見えない。判断基準が曖昧で、感情に流されやすい。

### 解決策：自分専用のダッシュボードを作る

「リベラルアーツ大学（リベ大）」の高配当株スクリーニング基準をベースに、自動でデータを収集・分析し、買い時を判断してくれるツールを作ることにしました。

**重要なポイント**：プログラミングの知識はありません。すべてAI（Claude）に指示を出して作ってもらいました。

## 第2章：4日間の制作プロセス

### Day 1（4月10日）：データの土台づくり

**やったこと**：
- Googleスプレッドシートからポートフォリオデータを取り込む仕組みを作成
- マネーフォワードの保有銘柄データをスプレッドシート経由で連携

**出した指示の例**：
「マネーフォワードに入っている保有銘柄のデータを、Googleスプレッドシート経由で取り込めるようにしてほしい」

**ポイント**：最初から完璧を目指さず、「まずデータを取り込めること」だけに集中しました。

### Day 2（4月11日）：画面と分析機能の構築

**やったこと**：
- ダッシュボード画面の作成（月別配当グラフ、セクター円グラフ、損益ヒートマップ）
- 株価の自動更新機能
- ポートフォリオ一覧表の作成

**出した指示の例**：
「保有銘柄の一覧を表で見たい。配当利回り、損益、セクターごとの割合もグラフで見たい」

**この日の気づき**：画面を見ながら「ここはもっとこうしたい」という具体的なイメージが湧いてきました。AIに見た目の改善を頼むときは「もう少し大きく」「色を変えて」のような日常的な言葉で十分でした。

### Day 3（4月12〜13日）：スクリーニングとデプロイ

**やったこと**：
- リベ大基準のスクリーニングエンジン開発（これが最大の山場）
- 買い時判断の自動ランキング生成
- 配当成長ポテンシャル分析の追加
- GitHub Pagesへの公開（スマホからも見られるように）
- 毎日深夜に自動更新する仕組み（夜中にPCが勝手にデータを取りに行く）

**出した指示の例**：
「リベ大の高配当株の選び方を基準にして、利回り3.75%以上、配当性向70%以下、連続増配している銘柄を自動でスクリーニングしてランキングにしてほしい」

「このツールをスマホでも見られるようにしたい。あと、毎日自動でデータを更新してほしい」

**スクリーニングの評価基準（11項目）**：
AIに伝えたのは「リベ大流の基準で判断してほしい」という大まかな方針だけです。具体的な11項目の評価ロジックはAIが提案し、自分が確認して「それでいこう」と決めました。

1. 配当利回り3.75%以上か？
2. 配当性向が70%以下か？（利益の何割を配当に回しているか）
3. 増配を続けているか？減配していないか？
4. 割安か？（PER×PBRのミックス係数で判定）
5. 売上は成長しているか？
6. EPS（1株利益）は伸びているか？
7. 営業利益率は10%以上か？
8. 自己資本比率は十分か？
9. 営業キャッシュフローは安定して黒字か？
10. 1株配当は成長しているか？
11. 利回りが高すぎないか？（罠銘柄チェック）

**データの信頼性**：
1つのサイトだけでは数値が間違っている可能性があるため、Yahoo!ファイナンス・株探・みんかぶ・IRバンクの4サイトからデータを取得し、クロスチェックする方針を取りました。

### Day 4（4月14日）：仕上げとUI改善

**やったこと**：
- ポートフォリオ表の列が多すぎる問題を解決（17列→9列デフォルト、ボタンで全列表示）
- ダッシュボードの不要セクションを削除（NISA枠管理、利回りアラート）
- 配当シミュレーションチャートの統合と操作性改善
- 「安定性」など分かりにくい表記の改善
- キャッシュフロー表記を記号（◎○△×）に変更

**出した指示の例**：
「ポートフォリオの列が多すぎて13インチのノートPCだと見切れる。重要な列だけデフォルトで表示して、残りは折りたたみにしてほしい」

「配当シミュレーションのグラフにカーソルを合わせても数字が出にくい。もっと反応しやすくしてほしい」

「安定性って書いてあるけど、意味が分かりにくい。増配回数と連続増配と減配回数があれば十分だから、安定性の列は消そう」

「CFが1/3、2/3とか書いてあるけど意味が分からない。◎○△×の記号にしてほしい」

## 第3章：ブラッシュアップの経緯 ── どう改善していったか

### 改善パターン1：「使ってみたら分かった」問題の修正

多くの改善は、実際に画面を見て操作してみて初めて気づいたものでした。

**例：ポートフォリオの合計行**
- 最初：合計行がテーブルの一番下にあった
- 問題：スクロールしないと合計が見えない
- 指示：「合計が一番上にある方が見やすいかも。検討してみて」
- 結果：合計行を一番上に固定表示し、スクロールしても常に見えるように改善

**例：詳細列の切り替えボタン**
- 最初：右上にボタンを配置
- 問題：目に入りにくい
- 自分の提案：「右下に固定でボタンをつけてみるのはどう？」
- AIの反応：「良いアイデアです。常に画面内に見えるので操作性が上がります」
- 結果：右下に半透明の固定ボタンを配置

### 改善パターン2：「それ、いらないかも」の判断

作った機能でも、使わないものは思い切って削除しました。

**例：NISA枠管理パネル**
- 当初は便利だと思って作った
- 実際には他のタブで確認できる情報と重複していた
- 判断：「いらないかな」→ 削除してダッシュボードをすっきりさせた

**例：安定性の列**
- 「安定」「変動あり」の2択だが、判定ロジックが単純すぎてミスリードの恐れがあった
- 隣に「増配回数」「連続増配」「減配回数」の具体的な数値があれば、安定性は自分で判断できる
- 判断：削除して列数を減らす方がメリットが大きい

### 改善パターン3：「分かりにくい」表現の改善

専門的な表記を、直感的に分かるものに変更しました。

**例：キャッシュフロー（CF）の表記**
- 変更前：「2/3」「3/3」→ 何の数字か分からない
- 変更後：◎（全年黒字）、○（概ね黒字）、△（やや不安定）、×（赤字多い）
- さらに色分け：◎は緑、○は青、△は黄色、×は赤

### 改善パターン4：技術的な問題の解決

AIが作ったコードにも不具合はあります。その都度、症状を伝えて修正してもらいました。

**例：チャートのツールチップが反応しない**
- 症状：「グラフにカーソルを合わせても数字が出にくい」
- 原因：マウスが正確にデータ点の上に来ないと反応しない設定だった
- 修正：カーソルがグラフ内のどこにあっても、最も近いデータ点の数値を表示するように変更

**例：データ取得が遅い**
- 症状：スクリーニングに10分以上かかる
- 修正：5つのデータを同時並行で取得するように変更 → 4分に短縮

## 第4章：完成したツールの全体像

### 機能一覧（8つのタブ）

1. **ダッシュボード**：保有資産の全体像（時価評価額、年間配当、セクター分布、配当シミュレーション）
2. **ポートフォリオ**：54銘柄の詳細一覧（損益、利回り、配当情報）
3. **配当カレンダー**：月ごとの配当金スケジュール
4. **買い時判断**：個別銘柄が今買い時かどうかの総合判定
5. **注目銘柄**：スクリーニングで選ばれた有望銘柄ランキング
6. **チェックリスト**：個別銘柄の詳細分析シート
7. **お気に入り**：気になる銘柄のブックマーク
8. **設定**：データ取り込み・更新の管理画面

### 自動化されている仕組み

- **毎日深夜0時**：株価・配当データを自動更新 → GitHub Pagesに自動デプロイ
- **Macを起動するだけ**：ローカルサーバーが自動で立ち上がり、ブラウザでアクセス可能
- **スマホ対応**：GitHub Pages経由でどこからでもアクセス可能（PWA対応）

### 技術スタック（参考情報）

- フロントエンド：HTML + CSS + JavaScript（フレームワーク不使用、1ファイル完結）
- バックエンド：Python（Flask的な軽量サーバー）
- データ取得：Yahoo!ファイナンス、株探（kabutan）、みんかぶ、IRバンクからスクレイピング
- デプロイ：GitHub Pages（docs/フォルダ経由）
- 自動化：macOS LaunchAgent（3つの常駐タスク）

## 第5章：AIへの指示の出し方 ── 非エンジニアのためのコツ

### コツ1：最初はざっくり、だんだん具体的に

最初の指示は「高配当株を管理するダッシュボードを作りたい」くらいで大丈夫です。AIが提案してきたものを見て、「ここをこう変えたい」と具体的に伝えていきます。

### コツ2：画面を見ながら「気になる点」を伝える

「13インチだと列が入りきれない」「このボタン、目に入りにくい」「カーソルの反応が悪い」など、使っていて感じたことをそのまま伝えるだけで改善されます。プログラミングの言葉に翻訳する必要はありません。

### コツ3：「これいるかな？」と疑問を持つ

作った機能が本当に必要かを定期的に見直します。「利回りアラートは他で見れるから、いらないかも」「NISA枠管理も別のところでカバーできている」。こうした引き算の判断が、ツールをシンプルに保つ鍵です。

### コツ4：AIに意見を聞く

「視認性どう思う？」「この構成、意見ある？」とAIに聞くと、改善提案をしてくれます。採用するかどうかは自分で決めますが、新しい視点が得られることが多いです。

### コツ5：分からないことは素直に聞く

「安定性って何？」「CFの1/3って何の意味？」と聞けば、ロジックを説明してくれます。説明を聞いた上で「分かりにくいから変えよう」と判断することもあります。自分が分からないということは、他の人も分からない可能性が高いです。

## 第6章：まとめ ── 4日間で学んだこと

### プログラミングができなくても、ツールは作れる

必要なのは「何を解決したいか」という課題意識と、「こうなったらいいな」というイメージです。技術的な実装はAIが担当します。

### 完璧を目指さず、使いながら改善する

Day 1の段階ではデータを取り込めるだけ。Day 4には54銘柄を自動分析する本格的なツールになっています。最初から完成形を設計するのではなく、「作る → 使う → 気づく → 直す」のサイクルを高速で回したことが成功の要因です。

### 「引き算」も大事な改善

機能を追加するだけでなく、不要な機能を削除する勇気も重要です。NISA枠管理、利回りアラート、安定性の列、別々だったシミュレーションチャート。いずれも「あった方がいい」と思って作りましたが、実際に使ってみると重複や分かりにくさの原因になっていました。

### AIは「相談相手」として使える

単にコードを書かせるだけでなく、「この構成どう思う？」「意見ある？」とフィードバックを求めることで、より良いツールになりました。最終的な判断は自分がしますが、AIの提案が改善のきっかけになることは多いです。
